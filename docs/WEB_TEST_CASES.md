# SmartRoute Web 端测试样例与问题记录

更新时间：2026-06-03

## 怎么给测试样例

建议每条样例都按下面格式给，方便队友复现、判断优先级，也方便之后转成 issue：

```text
入口：
输入：

预期：
实际：
影响：
优先级：
备注/截图：
```

如果是路线质量问题，建议额外记录：

```text
生成路线：
不合理点：
希望规则：
```

## P0/P1 问题

### 1. 问小团路线意图识别过低，导致不触发 SmartRoute

入口：问小团

输入：

```text
中国美术馆附近逛街、吃饭
```

预期：

- 应识别为路线/本地生活规划意图。
- 应触发 SmartRoute 插件。
- slots 至少识别出地点“中国美术馆附近”和需求“逛街、吃饭”。

实际：

- `/api/route-intent` 返回 `normal_answer`。
- confidence 为 `0.25`。
- detected_slots 中 location/time/constraints 基本为空。
- reason 为“当前问题更适合由小团普通问答处理”。

影响：

- 很典型的“地点 + 活动 + 餐饮”请求无法进入路线规划。
- 用户会感觉问小团没有理解当前问题。

优先级：P0

建议：

- 将“附近 + 逛/吃/喝/玩/看展/购物/半天/几个小时”等组合列为强路线意图。
- 不要求用户显式说“路线/规划/安排”才触发。
- 置信度规则应优先看当前 query，再参考画像。

### 2. 当前 query 被默认城市/画像覆盖

入口：API 直测 `/api/plan`

输入：

```text
中国美术馆附近逛街、吃饭
我下午想在中国美术馆附近逛两个小时，想看展、喝咖啡、吃晚饭，少走路
我下午要去深圳大学附近玩3个小时，帮我规划一个路线
外滩附近半天路线，不想排队，要有正餐，不要连着喝咖啡
```

预期：

- 北京样例应优先围绕中国美术馆/北京。
- 深圳样例应围绕深圳大学。
- 外滩样例应围绕外滩。
- 当前 query 中的城市、地标、品类、约束应高于默认画像偏好。

实际：

- 多条不同城市/地标输入都被解析成默认上海黄浦/徐汇路线。
- preferred_categories 固定包含景点、餐饮、咖啡/茶饮、购物。
- trace 中显示“读取画像：文艺体验型，近期偏好咖啡、展览和设计感空间”。

影响：

- 当前问题的地点和约束没有被可靠执行。
- 容易出现“北京问题生成上海路线”的严重体验问题。

优先级：P0

建议：

- current query 的 city/anchor/place/category/avoid 规则优先级高于画像。
- 如果地点解析失败，应该展示“地点解析失败/需要补充城市”，不要静默回退到默认上海。
- 画像只能做排序加权，不能覆盖显式需求。

### 3. 路线结构不合理：短路线中出现重复业态

入口：路线规划

输入：

```text
中国美术馆附近逛街、吃饭
```

实际路线示例：

```text
梧桐下甜品店（南京东路店）[咖啡/茶饮]
-> 转角茶饮实验室（南京东路店）[咖啡/茶饮]
-> 记忆美术馆（南京东路店）[景点]
```

另一个路线示例：

```text
寻味烧鸟（豫园店）[餐饮]
-> 南里烧鸟（南京东路店）[餐饮]
-> 老街展馆（徐家汇店）[景点]
```

预期：

- 2-4 小时路线中通常不应连续安排两家咖啡馆或两家正餐。
- 如果用户没有明确要求“探店/咖啡路线/美食路线”，路线应保证功能多样性。

影响：

- 路线不像真实可执行的一次出行。

优先级：P1

建议：

- 增加路线 composition 规则：短路线同一大类最多 1 个，除非 query 明确要求。
- 餐饮类区分“正餐/小吃/咖啡/甜品”，不要都按泛餐饮处理。
- 加入“主活动 + 正餐/咖啡补给 + 可选购物/散步”的模板。

### 4. 调整功能能返回结果，但没有按指令调整

入口：路线底部“调整”

基于原路线：

```text
咖啡/茶饮：梧桐下甜品店（南京东路店）
咖啡/茶饮：转角茶饮实验室（南京东路店）
景点：记忆美术馆（南京东路店）
```

调整输入：

```text
加一家餐厅，少安排咖啡馆
不要两家咖啡馆，换成一家正餐
少走路，餐厅优先
```

预期：

- 应减少咖啡/茶饮站点。
- 应替换或新增正餐类 POI。
- 调整摘要应解释“为什么换成餐厅/为什么保留某站”。

实际：

- 后端返回 `adjustment_status=applied`。
- 但实际替换的是景点，咖啡/茶饮仍然保留两家。
- 摘要偏向“替换高等待站点/减少等待”，没有遵守“加餐厅/少咖啡馆”。

影响：

- 用户会认为调整功能不可用或答非所问。

优先级：P0/P1

建议：

- 调整意图分类需要明确支持 add/remove/avoid/replace-category。
- `少安排咖啡馆` 应被识别为 avoid_categories 或 reduce_category。
- `加一家餐厅/正餐` 应被识别为 must_include_category。
- 调整后的 route validator 要检查原指令是否被满足，否则状态不应返回 applied。

## P1/P2 问题

### 5. “喜欢”和“不合适”按钮选不了

入口：路线结果卡片

预期：

- 点击“喜欢”后应有选中态/反馈 toast/写入画像记忆。
- 点击“不合适”后应有选中态，最好弹出不合适原因或记录负反馈。

实际：

- 用户反馈当前选不了。

影响：

- 无法验证个性化记忆闭环。
- README 中提到“喜欢/不合适会写入 SQLite 用户画像”，但 Web 端感知不到。

优先级：P1

建议：

- 前端加 selected/loading/success/error 状态。
- 后端 `/api/feedback` 返回写入结果和下一步影响说明。
- UI 上显示“已记住/下次少推荐同类”等反馈。

### 6. POI 详情页入口场景动机不够清楚

入口：POI 详情页

当前疑问：

- 用户为什么会在单个 POI 详情页发起路线规划？
- 是“围绕这个 POI 安排半日游”，还是“把这个 POI 加入现有路线”？
- 是从餐厅详情页延展，还是从景点详情页延展？

建议补充场景：

- 餐厅详情页：吃饭前后顺路去哪儿，等待期间去哪儿，饭后散步/咖啡。
- 景点详情页：景点前后如何安排，附近正餐/咖啡/购物。
- 收藏 POI：基于已收藏点串成半日路线。

优先级：P2

## 建议补充的测试样例

### 路线意图识别

```text
中国美术馆附近逛街、吃饭
国贸附近晚上和朋友吃饭逛街，预算200以内
深圳大学附近玩3小时，少走路，要有晚饭
外滩附近半天路线，不想排队，要有正餐
我在三里屯，等位40分钟，附近能先去哪儿逛一下
南山博物馆看完展，附近找个晚饭和咖啡
```

### 非路线意图

```text
帮我写一段美团产品介绍
SmartRoute 的产品定位是什么
中国美术馆几点开门
```

### 调整指令

```text
不要两家咖啡馆，换成一家正餐
加一个适合拍照的展览
少走路，地铁/打车优先
预算控制在人均150以内
不要排队超过15分钟
把第二站换成更适合亲子的地方
```

### 路线质量

```text
2小时：最多 2-3 个点，不能太分散
3-4小时：主活动 + 补给 + 正餐/购物
半天：允许 3-4 个点，但同类业态不要重复
明确“咖啡探店/美食探店”时才允许同类重复
```
